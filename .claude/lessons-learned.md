### 2026-06-21 ‚Äî Added `allowed-tools: Read, Grep, Glob` to three audit/bootstrap skills calling them "read-only"; their bodies document Edit/Write, and allowed-tools does not restrict anyway

- **Cause-tag:** `allowed-tools-semantics`
- **Symptom:** Recommending the optional `allowed-tools` lever, I added `allowed-tools: Read, Grep, Glob` to `auditing-glossary`, `auditing-claude-md`, and `bootstrapping-glossary`, characterizing them as "read-only audit/discovery" skills. A Layer-2 validation run then FAILed `auditing-glossary` claiming the grant "blocks" its documented correction step ‚Äî and the validator was *also* wrong (it read allowed-tools as restrictive). Both errors traced to the same field. The three skills' bodies in fact mandate mutation: "report the drift, **then correct it**" / "**Apply the corrections** ‚Äî a surgical edit" (auditing-*) and "**Create** the two foundational rules" (bootstrapping-glossary) ‚Äî they are not read-only.
- **Root cause:** Two compounding misreads of one frontmatter lever. (1) I assigned `allowed-tools` from an *expected* skill nature ("an audit skill is read-only") without grepping the body, where the documented Edit/Write steps were plain. (2) The grant-vs-restrict semantics: per `frontmatter-reference`, `allowed-tools` AUTO-GRANTS the listed tools without a prompt but does **not** restrict the rest ‚Äî Edit/Write stay callable. So a "read-only" list on a mutating skill is neither a safety restriction (it restricts nothing) nor a full friction win (the mutation step still prompts); it is just a mischaracterization. `spec-drift-audit` (genuinely read-only ‚Äî "report, don't fix") is the only one where a read-trio + `Bash` grant is accurate.
- **Wrong approach:** Picking `allowed-tools` from the skill's assumed role and labelling the skill "read-only" on that basis, without reading the body for documented mutation steps or checking what `allowed-tools` actually does.
- **Correct approach:** Treat `allowed-tools` as auto-grant-not-restrict. Before adding it, grep the skill body for documented mutation steps (`correct|fix the|apply the correction|surgical edit|create the|write the|edit the`). If present, the skill is not read-only ‚Äî either include the mutating tools in the grant (accepting that auto-granting Edit/Write on governance docs removes a human pause) or omit `allowed-tools` entirely; never describe such a skill as read-only.
- **Prevention:** When proposing or reviewing an `allowed-tools` value, run `grep -niE 'correct it|apply the correction|surgical edit|create the|write the|edit the' <skill>/SKILL.md` first; any hit means a read-only tool list is wrong for that skill. State the semantics inline (auto-grant, does not restrict) so a "read-only allowed-tools" claim is never mistaken for a capability fence.

### 2026-06-21 ‚Äî A prior task in the same plan wrote the new convention into the file later used as the RED control, so the baseline already carried the answer

- **Cause-tag:** `red-scenario-contamination`
- **Symptom:** Adding the `assets/` vs `references/` criterion to `writing-skills` (a methodology edit, Iron-Law-gated by RED‚ÜíGREEN). The RED subagent ‚Äî constrained to read only `writing-skills/SKILL.md` + `vocabulary.md` ‚Äî said "the methodology states no rule" yet still routed a template/prompt to `assets/`, because an *earlier task in the same plan* had already repointed a link to `assets/validation-subagent-prompt.md` inside that very SKILL.md. The subagent pattern-matched the leaked example, so the RED "failure" was not skill-free; a GREEN against it would have proved nothing.
- **Root cause:** Execution order. The plan moved files + repointed pointers (Task 1) *before* RED-testing the methodology edit (Task 3) ‚Äî and the methodology file is both the artifact-under-test and the RED control. Task 1 mutated that control to contain the new convention, so the "without-the-criterion" baseline already exhibited the convention. This is the same class as the existing `red-scenario-contamination` (the control carries the answer ‚Üí false signal), but the contaminating channel is the **shared tree/state mutated by a prior task**, not the scenario prompt's wording.
- **Wrong approach:** Trusting the RED because the *prompt* was clean (named no tool, constrained the files), without checking that the *files themselves* were free of the convention under test. Treating "subagent complied" as a valid baseline when a sibling task had already seeded the answer into the read set.
- **Correct approach:** Re-classified honestly ‚Äî the real reproduced failure here is **shaping** (the pre-existing placement variance: `subagent-dd` prompts lived in the skill root while peer prompts sat in `references/`), not discipline. GREEN was validated by the subagent **quoting the new rule passage** (not the example/inference) and correctly splitting a third file (a playbook ‚Üí `references/`) the example could not settle.
- **Prevention:** When a multi-task plan has an early task that writes a new convention into a file that a later task uses as a skill-edit RED control, **capture the RED before the contaminating task runs**, or run the RED control against a pristine copy of the methodology text (convention example stripped) ‚Äî never against the live, already-mutated tree. Pre-flight: before trusting any RED, grep the control's *read set* (not just the prompt) for the convention/symbol under test; any hit is contamination.

### 2026-06-21 ‚Äî Externalized a shared registry referenced by 4 skills into one skill's references/ (then into a rule); owner: a skill must be self-contained

- **Cause-tag:** `skill-self-containment`
- **Symptom:** A placeholder-key registry needed by 4 generator/auditor skills was placed in ONE skill's `references/` (`bootstrapping-claude-md/references/placeholder-keys.md`) and cross-linked from the other 3. The owner flagged the cross-skill links as unwanted coupling. My first fix ‚Äî move it to `.claude/rules/common/` as a shared rule ‚Äî was ALSO wrong: the owner pointed out a skill is a self-contained methodology meant to be droppable into any repo, and a rule (or a sibling skill's `references/`) may be ABSENT in the drop-target, and rules can change. Both placements break the skill's self-containment; the link resolves in the vault but dies when the skill ships alone.
- **Root cause:** The DRY instinct to centralize shared material into one file pulled me to externalize the registry ‚Äî first into a sibling skill, then into a rule. Both create a dependency on a path OUTSIDE the skill that is not guaranteed to exist where the skill is used. An agnostic skill's value is that dropping it unchanged into any repo preserves its function (`agnostic-skill-authoring`); a link to `.claude/rules/**` or `../<other-skill>/references/**` violates that exactly as a stack/path leak would.
- **Wrong approach:** Treating a multi-skill-shared artifact as a single source of truth to centralize (in a rule or one skill's references/) and cross-link ‚Äî optimizing DRY over the skill-is-droppable invariant.
- **Correct approach:** Duplicate the shared artifact into EACH skill's OWN `references/`, so every skill is self-contained and links only inside itself. Accept the controlled duplication; rely on the matching `auditing-*` skills + `dogfood-generator-sync` to keep the copies in sync (drift between copies is exactly what the auditors catch).
- **Prevention:** Before placing an artifact that more than one skill references, ask: "if this skill (and its `references/`) were dropped alone into a consumer repo, would this still resolve?" If it points to `.claude/rules/**` or another skill's `references/**`, the answer is NO ‚Äî duplicate the artifact into each skill's own `references/` instead of centralizing. Grep a finished skill AND its references for BOTH leak forms: outbound links (`grep -rnoE '\]\([^)]*\)' <skill-dir>` ‚Äî any `../<other>/` or `.claude/rules/` target) AND **prose mentions of a vault rule/skill name** (`grep -rn 'agnostic-skill-authoring\|dogfood-generator-sync\|<other rule/skill names>' <skill-dir>`). A naked prose reference to a rule (no markdown link) is the SAME self-containment leak ‚Äî the rule won't exist in the drop-target. State the principle inline instead of citing the rule by name.

### 2026-06-20 ‚Äî Over-read two AskUserQuestion rejections as a ban on the picker FORMAT and reverted to prose; the user wanted the picker

- **Cause-tag:** `overgeneralized-rejection`
- **Symptom:** Early in a session I presented two `AskUserQuestion` pickers (a disposition choice, then a scope choice). Both were rejected via the harness clarify/interrupt path. I treated those rejections as a standing signal that the user disliked the picker **mechanism** and switched all subsequent grilling questions to prose ("you rejected pickers earlier so I held back"). The user then had to explicitly correct me twice ‚Äî "–ø–æ—á–µ–º—É —Ç—ã –Ω–µ –¥–∞—ë—à—å —ç–ª–µ–º–µ–Ω—Ç—ã –Ω–∞ –≤—ã–±–æ—Ä" and "—Ç—ã –Ω–µ –¥–æ–ª–∂–µ–Ω –æ—Ç–∫–ª–æ–Ω—è—Ç—å —Ç–∞–∫–æ–π —Ñ–æ—Ä–º–∞—Ç –∫–æ—Ç–æ—Ä—ã–π —Ç—ã —É—á—É–¥–∏–ª" ‚Äî to get the selectable-options UI back. The objection had been to those specific questions' content/timing (premature disposition pickers fired before the design was shaped), never to the format itself.
- **Root cause:** Over-generalized instance-level feedback into a global behavioral ban. A rejected/interrupted tool call carries information about *that call* (its content, timing, or scope), not a verdict on the tool or output modality. The harness guidance is "a denied call means the user declined it ‚Äî adjust, don't retry verbatim"; I read "adjust" as "abandon the modality" rather than "fix the content and keep the form".
- **Wrong approach:** Inferring a durable preference ("user dislikes pickers") from one or two rejections and silently dropping the mechanism for the rest of the session, without checking whether the *format* or the *content/timing* was the objection.
- **Correct approach:** Scope a rejection to its instance: change what was wrong (here, stop firing pickers prematurely / fix the question content) but keep using the format unless the user signals the format itself is the problem. If genuinely unsure whether the form or the content was rejected, keep the form and ask, rather than assuming the broader ban.
- **Prevention:** When a tool use or output format is rejected/interrupted, before changing future behavior name the narrowest thing that plausibly triggered the rejection (content / timing / scope) ‚Äî default to that, not to dropping the tool or modality. Do not generalize a single rejection into "the user dislikes X format" without an explicit signal; a repeated steer back toward the format (a user asking "why no options?") is the tell that the over-generalization already happened.

### 2026-06-20 ‚Äî Tried to delete dead hooks and clean settings.json during an audit-fix; guard hooks blocked it and deleting the source first stranded the symlinks

- **Cause-tag:** `guarded-path-cleanup`
- **Symptom:** Applying audit findings, I ran `rm` on the dead `hooks/quality/{lint-fix,test-quick}.sh` plus their `.claude/hooks/` symlinks ‚Üí `security-guard.sh` (PreToolUse Bash) returned `BLOCKED: Attempt to modify security hooks or settings`. Splitting it, the source-file `rm` (under `hooks/quality/`) succeeded, but the symlinks under `.claude/hooks/` stayed blocked ‚Äî leaving two **dangling symlinks** I could no longer clean from either side. An `Edit` to `.claude/settings.json` (drop a stale `pnpm` allow-list) was likewise refused by `edit-write-guard.sh` (`Cannot modify security hooks, settings, or sensitive config files`).
- **Root cause:** Assumed normal Read/Edit/Write + Bash autonomy extends to the harness's own control files. It does not: `security-guard.sh`/`edit-write-guard.sh` deliberately fence off `.claude/hooks/**` and `.claude/settings.json` from the agent. Compounding it, I deleted the hook **source before the symlink** ‚Äî but the symlink path is itself guarded, so source-first ordering guarantees a stranded dangling link rather than a clean removal.
- **Wrong approach:** Treating `.claude/hooks/**` / `.claude/settings.json` cleanup as ordinary file ops, and deleting a guarded hook's source first on the assumption the symlink removal would follow.
- **Correct approach:** Any create/delete/edit under `.claude/hooks/**` or to `.claude/settings.json` is agent-blocked by the guards ‚Äî route the whole operation to the human as a single `!`-run command, and do NOT touch the source until the symlink can go in the same human step (deleting source-first only strands the symlink). The agent CAN still `Write` a throwaway script for the human to run (script *authoring* is not guarded; only running guarded paths is).
- **Prevention:** Before editing/deleting anything under `.claude/hooks/**` or `.claude/settings.json`, stop and hand it to the human as a one-line `!` command (or `bash scripts/<x>.sh`) ‚Äî never attempt it via Bash/Edit yourself, and never delete a guarded hook's source before its symlink is removed in the same step.

### 2026-06-20 ‚Äî AskUserQuestion failed twice with "expected array, got string"; the real cause was a `\x` escape in the JSON args

- **Cause-tag:** `tool-arg-json-escape`
- **Symptom:** Two `AskUserQuestion` calls were rejected with `InputValidationError: questions type is expected as array but provided as string`, even though `questions` was passed as a JSON array. The misleading message pointed at the type of `questions`; the actual fault was that option `description` strings contained `\xAB`/`\xBB` escapes (intended as guillemets). JSON permits only `\uXXXX`, never `\xNN`, so the args failed to parse and the harness surfaced a confusing top-level type error instead of an escape error. Using plain quotes / `\uXXXX` fixed it on the third try.
- **Root cause:** Hand-authored JSON tool arguments used a non-JSON escape (`\x`). The validation error names the consequence (a field that "isn't an array") rather than the cause (an invalid string escape upstream), so the message actively misdirects toward restructuring the array.
- **Wrong approach:** Reading "expected array, got string" literally and re-checking the array structure ‚Äî already correct ‚Äî instead of inspecting the string values for an invalid escape.
- **Correct approach:** In any tool's JSON arguments, encode non-ASCII only as `\uXXXX` (or paste the literal UTF-8 character); never `\xNN`, `\0`, or other C-style escapes JSON does not define. When a tool reports a type error for a field you clearly passed correctly, suspect an invalid escape in a sibling string value before touching the structure.
- **Prevention:** Before sending an `AskUserQuestion` (or any tool) call whose JSON contains typographic/Cyrillic characters, scan the argument strings for `\x`, `\0`, or other non-JSON escapes; use `\u` or the literal character. Treat a "field X is the wrong type" error on an obviously-correct field as a parse-failure symptom ‚Äî look for a bad escape, not a structural bug.

### 2026-06-19 ‚Äî Merging two quality hooks, ran lint + test identically on every edited file; owner flagged that a test runner is keyed to file role

- **Cause-tag:** `conflated-distinct-checks`
- **Symptom:** Consolidating `lint-fix.sh` + `test-quick.sh` into one `quality.sh`, the `.ts/.js` branch ran `eslint --fix` and `jest --findRelatedTests` identically on every edited file ‚Äî lumping two DISTINCT checks onto one trigger. The owner corrected it: lint and test are different checks with different conditions, and `jest` is keyed to the file's role ‚Äî tests live in `.test`/`.spec` files, so a test file should be run directly (`jest <file>`) while a source file runs only the tests RELATED to it (`--findRelatedTests`). My version applied `--findRelatedTests` even to a test file (wrong) and treated lint+test as one condition.
- **Root cause:** I treated "merge two hooks" as "concatenate their commands under one matcher" and preserved "run both on every match," instead of re-deriving each check's real trigger. Lint (per touched file, role-agnostic) and test (role-dependent: run-self vs run-related) have genuinely different conditions; collapsing them onto one branch erases that.
- **Wrong approach:** Merging distinct checks by stacking their commands on one condition because they previously shared one PostToolUse matcher.
- **Correct approach:** When merging/wrapping distinct checks, give each its own branch keyed to its own condition. For a test-runner wrapper specifically, branch on file role: `*.test.*`/`*.spec.*` ‚Üí run the file directly; a source file ‚Üí related-tests mode; lint runs on any touched file independently. (Verified with a fake-`npx` shim: source file ‚Üí `eslint` + `jest --findRelatedTests`; test file ‚Üí `eslint` + `jest <file>`.)
- **Prevention:** Before consolidating multiple checks into one hook/script, list each check's actual trigger condition and give each its own branch ‚Äî never assume two checks share one condition just because they shared one matcher/trigger. For any test-runner wrapper, branch test-file vs source-file role rather than running it like the linter.

### 2026-06-19 ‚Äî Behavioral subagent RED/GREEN to validate a structural-defect fix (hallucinated skill-name + leak in `handoff`) false-GREENed; the RED was the wrong instrument

- **Cause-tag:** `red-instrument-mismatch`
- **Symptom:** To behaviourally prove the `handoff` #1 fix (removed a `pnpm`/`src/shared` stack leak, the hallucinated `scaffold-slice`/`scaffold-api` skill names, and a false "`skill-gate.sh` will deny" claim) I ran two cold subagents on a Rust consumer scenario ‚Äî one with the OLD body, one with the NEW. BOTH complied: the OLD-body agent ignored the leaked `pnpm`/`scaffold-*` and substituted `cargo check`/Rust layers on its own; the NEW-body agent did the same. RED did not reproduce ‚Üí a clean run that proves nothing (framework.md Suspicion #4). Worse, "RED shows no failure" reads under the Iron Law as "nothing to fix" ‚Äî which would wrongly cast doubt on a fix whose defect was already proven *statically* (`find` shows no `scaffold-*` skill; `grep` shows the stack tokens).
- **Root cause:** Wrong RED *instrument* for the failure class. A hallucinated-skill-name / agnostic-leak / broken-reference defect is STRUCTURAL ‚Äî decidable by `find`/`grep` (a cited skill/path is absent, or a stack token is present). A behavioral subagent RED cannot exert that pressure: a capable model substitutes the real stack, and an in-vault subagent additionally inherits framework.md's agnostic doctrine (the `scoping-skill-value.md` contamination caveat), so both runs comply regardless of the skill body. `scoping-skill-value.md` frames the contamination for *discipline*-RED only and, read literally ("if you cannot reproduce the failure, write nothing"), would even misguide toward NOT fixing a real structural defect ‚Äî it does not handle this class.
- **Wrong approach:** Validating a structural fix behaviourally, and treating a non-reproducing in-vault RED as evidence about whether the defect exists at all.
- **Correct approach:** For hallucinated-name / agnostic-leak / broken-reference classes the RED *is* the static check ‚Äî `find skills -name SKILL.md` / `grep` proving the cited skill/path is absent or the stack token present; GREEN is re-running that grep clean + confirming links resolve. Reserve behavioural subagent RED/GREEN for discipline / shaping / efficacy classes (discipline in a clean, non-vault env). A clean behavioural RED never licenses skipping a statically-proven fix.
- **Prevention:** Before running a behavioural subagent RED to "prove" a skill fix, classify the defect first. Structural (a symbol / skill-name / path / token a `grep` or `find` can decide) ‚Üí verify with `find`/`grep`; do NOT run a behavioural RED, and never read its non-reproduction as "nothing to fix". Discipline / shaping ‚Üí behavioural RED (discipline in a clean env). If both an OLD-body and a NEW-body run comply, the scenario isn't exerting the pressure (Suspicion #4) ‚Äî switch instruments, don't trust the GREEN.

### 2026-06-19 ‚Äî A relocate/rename spec + drift-audit reported "zero drift" while 4 concept-level references were still stale (false GREEN)

- **Cause-tag:** `relocate-reference-undercount`
- **Symptom:** Relocating `domains-glossary.md`/`framework.md` from `rules/common/` ‚Üí `rules/domains/`, the spec's reference set and the audit's verification grep (`rules/common/\(framework\|domains-glossary\)`) both passed ‚Äî I declared "zero drift / complete". A cold reviewer's independent broad grep then found 4 still-stale references: root `CLAUDE.md:19` (prose "rules/common/ (framework + domain glossary)"), `CLAUDE.md:51` ("Where rules live" table row), `bootstrapping-domain-rules/SKILL.md:13` (taught convention "under .claude/rules/common/"), and `root-claude-md-template.md:75` (agnostic "Where rules live" row). All describe the moved concept's home via the **bare directory token** or the **concept name**, none via a `dir/file.md` link.
- **Root cause:** I derived the reference set ONE way ‚Äî literal path-links to the moved files (`rules/common/framework.md`) ‚Äî and scoped the verification grep to `dir/(fileA|fileB)`. That pattern structurally cannot match prose/tables that name the home as bare `rules/common/` or by concept ("framework + domain glossary"). The gate could not fail on the very references the change existed to fix ‚Üí false GREEN (framework.md Suspicion Protocol #4, test-passes-for-wrong-reason). A relocate also has a *conceptual* surface (docs that describe where the thing lives), not just a link surface.
- **Wrong approach:** Treated "all references" as "all markdown links to the file path", and wrote the audit grep narrow enough to green on the partial change.
- **Correct approach:** Derive references TWO ways and gate on the broad one: (1) literal path-links to the moved/renamed artifact, AND (2) the bare directory/path token PLUS the concept name in prose and tables. Write the verification grep on the BARE token (`grep -rn 'rules/common/' ‚Ä¶`) and filter the items that legitimately stay, rather than enumerating the specific files that move.
- **Prevention:** For ANY relocate-or-rename change, before declaring done run `grep -rn '<old-bare-dir-or-token>'` across the live tree (not `<dir>/<specific-file>`), subtract the known legitimate stayers, and confirm the remainder is empty; additionally grep the moved thing's **concept name** in prose/tables. A verification pattern that is narrower than the change's blast radius is a false gate.

Don't trust a clean structural-validator pass as behavioural proof ‚Äî RED a literal-reading scenario before calling it done.

- **Prevention:** When a skill/rule you are authoring asserts what a *neighbour* skill does (stamps X, consumes Y, traces Z), `grep`/`Read` that neighbour's `SKILL.md` for the named mechanism before writing the sentence; if grep returns nothing, the mechanism doesn't exist ‚Äî describe the hand-off without inventing the receiver's behaviour.

- **Cause-tag:** `human-handoff-command-fragility`
- **What happened:** The hook-migration plan had a `[HUMAN]`-run step: `git mv` 12 files + run the linker. I gave the human a multi-line, `\`-continued `&&` chain to paste after `!`. First paste fired only `git mv .claude/hooks` (continuation lost ‚Üí usage error); the single-physical-line `&&` retry hit `zsh: parse error near &&` (paste truncation again). Two wasted round-trips. ### 2026-06-19 ‚Äî `git mv` + symlink at the IDENTICAL old path registers as add+typechange, not a rename (history check needs a blob compare)

- **Cause-tag:** `git-symlink-shadows-rename`
- **What happened:** Verifying "history preserved" after the hook move, `git diff --cached --find-renames` returned ZERO renames and `git status` showed ` T .claude/hooks/<name>.sh` (typechange). Alarming ‚Äî the skills migration had shown clean `R100` renames. Root cause: for skills the moved unit was a *directory* and the symlink took a *new* leaf path, so the old file path was deleted ‚Üí git paired it as a rename. For hooks the symlink occupies the **exact same path** as the old file (`.claude/hooks/<name>.sh`), so the old path is not deleted ‚Äî it becomes a symlink (mode 120000) ‚Äî and git sees "new file added under `hooks/` + old path typechanged to symlink", which rename detection does not pair. History is still intact: `git rev-parse HEAD:.claude/hooks/<name>.sh` == `git rev-parse :hooks/<cat>/<name>.sh` (identical blob `3cee0a7‚Ä¶`), so `git log --follow` traces it by content.
- **Fix / rule:** "No `R` in `git status`" does NOT mean history was lost. When a migration places a symlink at the same path a real file used to occupy, verify history by **blob-identity** (`git rev-parse HEAD:<old>` == `git rev-parse :<new>`), not by rename detection. The end state (real file added at new path, old path staged as a 120000 symlink) is correct and committable.
- **Prevention:** For any "move + symlink-back" migration, pick the history check by shape: dir-move with symlink at a new name ‚Üí expect `R100`; file-move with symlink at the identical old path ‚Üí expect add+typechange, confirm via blob-equality instead.

### 2026-06-19 ‚Äî `writing-lessons`'s own promotion-debt scan silently matched nothing (format drift)

- **Cause-tag:** `self-check-format-drift`
- **What happened:** The scan command documented in `writing-lessons/SKILL.md` was `grep -oE 'Cause-tag\*\*:‚Ä¶'` ‚Äî it matches `Cause-tag**:` (template format), but the real entries are written `**Cause-tag:**` with the tag in backticks, and the file's intro prose says `` `Cause-tag` ``. Three cosmetic formats of one field had drifted apart and the self-check matched none of the actual entries.

- **Prevention:** Author/verify any "run this to self-check" command against the REAL artifact at write time, never assume it matches the template; for markdown field scrapes, anchor on the field's **line-start list marker** (not just the bold span, which prose can quote) and allow `[^[:alnum:]]+` between marker and value.

### 2026-06-19 ‚Äî A RED scenario that foregrounds tool choice contaminates the test and yields a false GREEN
