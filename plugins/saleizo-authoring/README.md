# Authoring Kit

Test-first authoring of the framework's own artifacts. Every skill here drives a RED → GREEN → REFACTOR → VALIDATE loop: you create a skill, hook, rule, lesson, or ADR only after a baseline check fails, then prove it holds.

## Quickstart

- Writing or changing a skill? Reach for `writing-skills`. A Claude Code hook? `writing-hooks`. A project rule? `writing-rules`.
- Capture a recurring failure as a lesson with `writing-lessons`; record an architectural decision with `writing-adrs`.
- Install it first — see [Installation](#installation).

## How it works

Each skill is invoked by naming the authoring task ("write a skill", "add a hook", "record this decision"). The skill supplies the methodology — the failing-test-first discipline, the validators, and the cold-reviewer gate — so the artifact is verified, not just written.

## Installation

### Claude Code

```text
/plugin marketplace add enkeapple/saleizo
/plugin install saleizo-authoring@saleizo
```

## What's Inside

- **writing-skills** — author, edit, or validate a skill in this repo test-first (RED → GREEN → REFACTOR → VALIDATE).
- **writing-hooks** — author or edit a Claude Code hook test-first (fixture-execution is its RED/GREEN).
- **writing-rules** — write or edit a project rule under `.claude/rules/`, or promote a recurring lesson into a durable rule.
- **writing-lessons** — capture a qualifying lesson from a session that exposed a reusable, recurring failure.
- **writing-adrs** — record an architectural decision as an immutable, gated ADR; supersede (never edit) when it changes.

## Philosophy

This repo exists to practice test-first authoring on itself. These skills are how that discipline is applied to skills, hooks, rules, lessons, and decisions — no artifact ships because the markdown exists; it ships because a baseline failed without it and a validated run confirms it.

## Contributing
