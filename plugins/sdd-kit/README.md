# Sdd Kit

The gated spec-driven-development chain — turn a fuzzy idea into shipped, spec-checked code one approved step at a time — plus the skills that author and maintain skills, hooks, and rules, and bootstrap the framework into a fresh repo.

## Quickstart

1. Install it — see [Installation](#installation).
2. Run the chain on a change: `grilling` (design) → `writing-specs` → `writing-plans` → `pre-implementation-protocol` → execute test-first (`test-driven-development`) → `spec-drift-audit`. Or drive the whole thing with `sdd-lifecycle`.
3. Short aliases for the common entries: `/grill`, `/spec`, `/sdd`, `/audit`.

## How it works

Each phase is its own skill and ends at an explicit approval gate — nothing auto-advances. A skill hands its artifact (design → spec → plan) to the next, so a weak hand-off between two steps is visible instead of silent. Authoring and setup skills sit alongside the chain for changing the framework itself.

## Installation

### Claude Code

```text
/plugin marketplace add enkeapple/obilyk-skills
/plugin install sdd-kit@obilyk-skills
```

## What's Inside

- **chain** — the gated SDD pipeline from design through drift audit, each phase test-first.
- **authoring** — write skills, hooks, rules, and lessons (each test-first).
- **setup** — bootstrap and audit a consumer repo's CLAUDE.md, glossary, and README.
- **session** — persist and resume work across a context limit (`handoff`).
- **aliases** — short user-typed entry points to the chain's common phases.

## Philosophy

- **Gates over momentum.** Every phase stops for explicit approval; the chain never races ahead of agreement.
- **Test-first, no exceptions.** No skill — and no skill edit — without a failing subagent run first.
- **Agnostic by default.** The chain bakes in no stack or paths; the consumer repo supplies them.
- **Surface the bottleneck.** A leaky hand-off or an over-rigid step is the bug, and it becomes the next lesson and rule.

## Contributing

Author or change a skill through `writing-skills` (RED → GREEN → REFACTOR → VALIDATE). A change is done only when the validators pass and an independent subagent run confirms the behavior. Regenerate this catalog with `bootstrapping-readme`; check it for drift with `auditing-readme`.
