# README Scaffold Example

Illustrative (a real run derives every row from disk; your repo name, plugins, skills, and categories will differ). Shows the full landing-page scaffold the skill emits **when creating a fresh README** — a superpowers-style page — and how the derived block sits inside the managed markers.

Two zones, with one hard boundary:

- **Human-owned scaffold prose** (everything OUTSIDE the markers — intro, Quickstart, How it works, Installation, What's Inside, Philosophy, Contributing). Emitted ONCE as `<!-- TODO -->` placeholders for a human to fill; never regenerated on a re-run, never audited. Rephrase, reorder, or drop these freely.
- **Derived block** (everything BETWEEN the markers). Every row is derived per the [catalog-derivation contract](../references/catalog-derivation.md); regenerated on every re-run; audited by `auditing-readme`. Never hand-edit inside the markers.

On a re-run against an existing README, the skill replaces ONLY the managed block — the scaffold prose is left exactly as the human last left it.

## Per-plugin README (marketplace) / root README (single-repo)

A self-contained, superpowers-style page with a **skills-catalog** block. In a marketplace this is each `plugins/<name>/README.md`, and its skill links are **plugin-relative** (`skills/<category>/<name>/SKILL.md`, resolving from the plugin dir). In a single-repo this is the one root README, links repo-root-relative. The H1 is the Title-Case form of the plugin/repo `name` (`guardrails-kit` → `# Guardrails Kit`). The Quickstart uses anchor links into the Installation subsections; Installation carries an `###` per platform:


## Hooks-only plugin (marketplace) — hooks catalog

A plugin that ships only `hooks/` (no `skills/`) emits a **hooks-catalog** block and NO empty skills block. Its rows are derived from `hooks/hooks.json`, grouped by `###` event (alphabetical), each linking to the script **plugin-relative** (`hooks/<file>.sh`), the description from the script's line-2 header comment (or the `<event> hook (matcher: …)` fallback). The surrounding scaffold is the same superpowers-style page, owning the block under a `## Hooks` heading:

````text
# Guardrails Kit

<!-- TODO: one-paragraph intro — what this plugin guards and who it's for. Human-authored. -->

## Hooks

<!-- hooks:start -->
<!-- Generated by bootstrapping-readme — do not edit by hand; rerun the skill. -->

### PostToolUse

- **[detect-bypass](hooks/detect-bypass.sh)** — detect skill bypass.
- **[token-guard](hooks/token-guard.sh)** — estimate tokens from tool output size, warn/block on overuse.

### Stop

- **[friction-log](hooks/friction-log.sh)** — log DETERMINISTIC friction signals from the turn's transcript.

<!-- hooks:end -->
````

A plugin that has BOTH skills and hooks emits the skills block first, then the hooks block — each under its own `## Skills` / `## Hooks` heading.
